# v11.9 State-Sync Activation and Join Authorization

**Status:** the explicit, boot-only state-sync path is implemented but is not a
v11.9 release claim. With no `quorum.state_sync` role configured, the runtime
keeps the ABCI endpoints empty/reject/abort. An explicitly configured provider
or pristine receiver instead loads a strict local join authorization, hardens
the effective CometBFT peer profile, arms the real endpoint controller before
Comet starts, and admits ordinary serving only after the final application
bundle is sealed. The release workflow remains intentionally blocked on a real
Docker app-v20 signed scope-reconfiguration plus authorized state-sync transfer
test.

The deterministic latest-visible state stream, canonical fsynced activation
journal, pre-handshake recovery, isolated candidate verification with bounded
provider fallback, requester-anonymous serving endpoints, one-shot receive
session, whole-application finalizer, bounded Comet persistence/handoff/seal
wait, and final store/service rebinding are implemented. They remain opt-in
because the local authorization file is an operator-installed trust root and
because raw Internet validator transport still requires separately routable
TCP, port forwarding, or an operator VPN. A tunnel layer is separate future,
out-of-scope work and is not a v11.9 release gate. The integrated authorized
provider-to-receiver proof and signed app-v20 scope reconfiguration remain the
release blockers; none of these are implied by the endpoint substrate.

## 1. Boundary

State sync is a same-chain validator/bootstrap mechanism. It is not the v11.8
independent-chain synchronization overlay and it is not a general federation
download endpoint.

The network format contains a deterministic stream of the latest visible
Badger key/value set plus canonical metadata and chunk hashes. It is not a
Badger backup stream: physical LSM versions, tombstones, and expired values do
not cross the network; keys are emitted in strict bytewise order, and TTL or
user metadata fails export because dropping either could change future
behavior. Restore requires an empty database and rejects duplicate/out-of-order
keys, invalid record lengths, truncation, an empty stream, count mismatch, and
trailing bytes. The format must never contain SQLite/PostgreSQL projections,
CometBFT databases, node or validator keys, vault keys, local configuration,
logs, or binaries. The older operator rollback snapshot explicitly contains
local/private material and is never accepted by the network decoder.

Delivered substrate:

- `internal/statesync/canonical_state.go` — the versioned, bounded,
  latest-visible key/value stream and strict empty-database restore.
- `internal/statesync/format.go` — bounded canonical metadata, trusted height
  and AppHash binding, chunk hashes, and full-stream hash.
- `internal/statesync/assembler.go` — fresh-directory, retry-safe disk assembly.
- `internal/statesync/exporter.go` — verified canonical-state publication and
  fail-closed catalog/chunk loading, provider-root maintenance, and bounded
  retention.
- `internal/statesync/disk_space.go` — bounded receiver staging/preparation
  capacity preflights before large disk work begins.
- `internal/abci/scoped_state_sync.go` — fresh-directory restore, exact height,
  app-v20 audit, narrow AppHash, stored AppHash, and canonical scoped-state
  verification plus a read-only recovery inspector for fresh, older, and
  post-app-v20 live directories.
- `internal/abci/boot_state_sync_endpoints.go` — authorization-aware serving,
  a one-shot receive session, isolated preverification, bounded bad-snapshot
  fallback, latched receiver expiry, and approved-sender enforcement.
- `internal/statesync/activation.go` and
  `internal/statesync/activation_directories.go` — bounded canonical activation
  journal, atomic fsynced journal replacement, guarded quarantine/live renames,
  verification callbacks, symlink/path rejection, and fail-closed restart
  recovery against persisted Comet state. The post-bootstrap seal validates
  already-observed live app/Comet state and removes quarantine without reopening
  the live Badger handle. These paths do not start ordinary services.
- `internal/store/badger.go:105-128` — writable open of an existing verified
  state-sync database without constructor migrations/backfills.
- `cmd/sage-gui/state_sync_boot.go` — persisted Comet state reader and startup
  journal recovery invoked before the canonical Badger directory is created or
  opened.
- `cmd/sage-gui/state_sync_runtime.go` — production role arming, exact pristine
  receiver checks, Comet light-client configuration, and final-bundle
  preparation/activation wiring.

## 2. Why a live Badger rename is rejected

The current process distributes concrete pointers before and after CometBFT
starts:

- `SageApp` owns a concrete `*store.BadgerStore`, validator set, governance
  engine bound to that store, and cached app state
  (`internal/abci/app.go:254-261`, `internal/abci/app.go:1837-1854`).
- the local CometBFT client receives a `BootStateSyncRuntime` through the ABCI
  application interface (`cmd/sage-gui/node_controller.go`);
- after Comet bootstrap, startup seals any armed activation, durably cleans its
  directory transaction, then acquires and freezes the runtime's final
  application. Only then may the nonce allocator, projection rebuilder, local
  snapshot scheduler, upgrade watchdog, REST server, dashboard, or background
  workers capture concrete app/store handles (`cmd/sage-gui/node.go`).
- the local rollback scheduler retains the raw database handle, not a dynamic
  accessor (`internal/snapshot/snapshot.go:110-116`).

Renaming a prepared directory under these objects would leave some readers on
the old database and others on the new one. Mutating `BadgerStore.db` alone is
also unsafe: typed store methods call the field directly without a process-wide
lease, so a concurrent REST read or snapshot backup could race a close/swap.

Therefore activation must replace the complete consensus application bundle
during boot, before ordinary services are constructed. It must not hot-swap a
store inside an already serving `SageApp`.

## 3. Chosen runtime ownership model

`BootStateSyncRuntime` now implements the CometBFT ABCI application interface
and owns exactly one immutable bundle reference:

```text
ConsensusBundle
├── SageApp
├── BadgerStore
├── ValidatorSet
├── GovernanceEngine
├── AppState / fork caches
└── expected height + AppHash + app version
```

All normal ABCI methods take a short runtime read lease and delegate to the
current bundle. State-sync methods are implemented by the runtime itself, not
by the old `SageApp`, so the final chunk can replace the complete bundle without
closing an object that is still executing its own method.

The runtime is dormant by default and can be armed only once, before normal
serving acquires the application. It captures and validates the bundle's
`Info` height/AppHash/app version, rejects lower-height and same-height
conflicting replacements, holds an exclusive lease across complete-bundle
activation, and validates the replacement against the trusted snapshot. The
runtime owns the four state-sync endpoints itself rather than delegating them
to `SageApp` (`internal/abci/boot_state_sync_runtime.go`,
`internal/abci/boot_state_sync_endpoints.go`). Bundle cleanup owns only
consensus Badger state; the shared off-chain projection remains process-owned
across replacement.

The runtime accepts activation only in a one-shot boot phase. Once CometBFT has
persisted the synchronized state and the runtime is sealed, `OfferSnapshot`
and changed chunk retransmissions fail closed for the lifetime of that process.
A serving process cannot be converted into a receiver, and a receiving process
cannot accept a concurrent offer or a replacement after activation begins. A
candidate rejected during isolated preverification may return to discovery for
the bounded provider fallback described below.

While a receiver is in `discovering`, `assembling`, or `prepared`, the runtime
keeps `Info` and the four state-sync methods available to CometBFT but rejects
ordinary serving. `Query` returns
`ErrBootStateSyncConsensusServingBlocked`, while `CheckTx` returns ABCI code
`110` in codespace `sage-state-sync` so Comet's mempool client is not handed a
transport error. Both remain non-blocking throughout every unsealed phase,
including `activated_pending_comet`; allowing either to wait could hold
Comet's shared in-process ABCI-client mutex and deadlock the syncer's final
`Info` verification. Consensus methods (`InitChain`, proposal,
block-finalization, vote-extension, and `Commit`) fail fast before
`activated_pending_comet`. During that narrower phase, a consensus block call
may already have been started by Comet's successful handoff; it waits on the
seal/failure gate without holding the bundle lease. The call is delegated only
after `sealed`, or wakes with SAGE's seal-abort sentinel during
shutdown/failure. The local block-sync overlay treats only that direct or
wrapped sentinel as a graceful reactor exit; every unrelated application error
retains Comet's upstream panic behavior.

`SageNodeController` must hold the ABCI application interface/runtime rather
than a concrete `*SageApp`. After the runtime seals, startup obtains the final
bundle and only then constructs:

1. projection recovery and `/ready` state;
2. the local rollback snapshot scheduler with the final raw Badger handle;
3. nonce-floor and upgrade-watchdog callbacks;
4. REST, dashboard, MCP, federation, and background workers.

This serving-order boundary is delivered: projection rebuild, snapshot binding,
nonce-floor setup, FTS backfill, embedder/managed-sidecar startup, and every
later REST/dashboard/MCP/federation worker are behind the seal and final-bundle
acquire. Incomplete or failed phases tear Comet back down without starting
ordinary services. Freezing the bundle before concrete pointers escape removes
the need for dynamic store proxies in every handler.

## 4. Activation state machine

Only these transitions are valid:

```text
disabled
   └─> discovering ─> assembling ─> prepared ─> activated_pending_comet ─> sealed
          ^               |
          +── rejected ───+  (maximum eight candidates)

fatal error from any active phase ─> failed
```

- **disabled:** default; current fail-closed ABCI behavior.
- **discovering:** one trusted Comet offer is validated against format, height,
  metadata hash, light-client AppHash, authorization, and staging-disk
  capacity.
- **assembling:** chunks may arrive out of order; exact duplicates are
  idempotent. A changed duplicate, wrong size/hash, or unknown index requests a
  retry; a second offer aborts the one-shot attempt. After the complete stream
  hash verifies, the candidate is restored and inspected in an isolated Badger
  directory while the old live bundle remains open. A malformed prepared
  candidate is rejected and the runtime returns to `discovering`, allowing a
  different authorized provider snapshot; eight rejected prepared candidates
  exhaust the bounded fallback and fail the process-local attempt.
- **prepared:** isolated restore has already passed exact height, epoch,
  app-v20, narrow AppHash, persisted AppHash, and canonical scoped-state
  verification. Live state is still untouched and open at entry to this phase.
- **activated_pending_comet:** the verified directory is active in a newly
  constructed consensus bundle, but CometBFT may not yet have durably
  persisted the seen commit, atomically published positive-height state with
  its effective-height bridge, completed its block-sync handoff, and published
  the exact completion marker. Any early consensus block execution waits behind
  the seal; `Query` and `CheckTx` continue to fail fast.
- **sealed:** CometBFT's live persisted height/AppHash/app version exactly match
  the activated application bundle; the old Badger directory can be retired,
  and the final bundle can be frozen and handed to post-seal projection/
  readiness recovery. Ordinary services and workers may be constructed only
  after this boundary; readiness still waits for the projection checks below.
- **failed:** readiness remains false; no normal serving starts. A retry always
  creates new staging paths and cannot reuse unverified chunks.

No transition may lower the trusted height or change the trusted AppHash.
Fallback is permitted only from `assembling` to `discovering`, before the old
bundle closes or any activation journal/rename occurs.

## 5. Crash-safe filesystem transaction

Activation uses a small fsynced journal outside both Badger directories. The
journal contains only paths, phase, snapshot height, metadata hash, and trusted
AppHash; never keys or memory contents.

1. Before any live close, journal, or rename, preflight candidate-preparation
   capacity, restore the assembled canonical stream into a fresh isolated
   Badger directory, and verify exact height/epoch, active app-v20, narrow and
   persisted AppHash, and every canonical scoped envelope. A bad candidate is
   discarded and may fall back to another authorized provider snapshot.
2. Recheck the authorization deadline, enter `prepared`, acquire
   the runtime write lease, and wait for every delegated ABCI read lease to
   drain.
3. Close only the old consensus Badger handle. The shared off-chain projection
   is process-owned and must not be closed by bundle replacement.
4. Recheck authorization, write the `prepared` journal, and fsync the journal
   plus parent directory. Recheck again before each rename: live Badger moves
   to a unique quarantine path, then prepared Badger moves to the canonical
   live path, with the parent fsynced after each step. If authorization expires
   during these durable boundaries, the receiver latches expiry and rolls the
   directory layout back to the old live state wherever a durable step has
   occurred.
5. Open the received database with a new writable **no-backfill** constructor
   and reverify it under its promoted name.
   `NewBadgerStore` cannot be used because its upgrade backfills can mutate the
   just-verified AppHash (`internal/store/badger.go:45-82`).
6. Construct a fresh `SageApp`/validator set/governance engine, then re-check
   height, active app-v20 audit, and AppHash from the writable handle. Apply the
   same process-local application policy (including display version and block
   retention) as the initial bundle.
7. Publish the new immutable bundle and fsync
   `activated_pending_comet` before the final chunk returns `ACCEPT`.
8. CometBFT calls `Info` and verifies exact app version, height, and AppHash
   before it reports restore success. Its asynchronous completion path then
   synchronously persists the verified seen commit. `StateStore.Bootstrap`
   publishes positive-height state and its effective state-sync-height bridge
   in the same synchronous batch, then Comet successfully calls
   `SwitchToBlockSync` and synchronously records the exact handoff
   height/AppHash. The bridge remains durable across consecutive restarts while
   the block store is empty, allowing block sync to start at `H+1`; it is
   ignored once the block store has a real height. An H+2-eligible provider may
   already have catch-up blocks available. A resulting consensus block call
   waits on SAGE's PendingComet gate instead of returning an error to Comet's
   block executor, while `Query` and `CheckTx` continue to fail fast.
9. Startup polls the node-owned state/block stores under a bounded context and
   requires matching positive-height state, the cryptographically valid seen
   commit, and the exact post-switch completion marker. The default bound is 30
   minutes; an operator may set any positive `startup_timeout` (15–30 minutes
   is normal). Only then does SAGE
   write `sealed`, fsync, release waiting block execution, remove quarantine and
   journal, freeze bundle replacement, and publish concrete app/store pointers
   to normal services. Timeout or mismatch releases shutdown waiters and tears
   startup down fail closed.

On boot, journal recovery compares the persisted CometBFT height with the
snapshot height before opening SAGE Badger:

- Comet height below snapshot: restore the quarantined old Badger and retry
  state sync; the prior ABCI activation was never committed by CometBFT. Comet
  synchronously caches the completed genesis document before asynchronous
  state sync begins. Startup recognizes that state DB only when it is the sole
  record and is byte-identical to the live configured genesis. It separately
  recognizes the exact lone commit-first block-store record possible before
  state publication. Every persistence surface is inspected before either
  record is removed, so changed genesis, extra keys, malformed data, signing
  history, or WAL evidence is preserved and rejected without partial cleanup.
  Raw inspection also avoids constructing a `BlockStore` over malformed
  metadata. The two exact deletions are idempotent if another crash lands
  between them.
- Comet height at or above snapshot with matching trusted state: keep the new
  Badger, finish recovery, and turn off the recovered one-shot receiver for
  that process so normal block sync can resume.
- missing/ambiguous directories, equal height with a different trusted hash, or
  impossible phase combinations: fail closed and require operator recovery.

This recovery check must run before the ABCI handshake. Otherwise an app height
ahead of CometBFT can brick startup.

## 6. Internet join authorization

ABCI `ListSnapshots` and `LoadSnapshotChunk` do not receive the requesting peer
identity. Application code therefore cannot enforce a per-request allowlist.
Authorization must be established at the authenticated CometBFT P2P connection
boundary before snapshots are advertised.

State-sync serving requires a **validator-only P2P profile**:

- PEX and seed mode are false, seeds are empty, and there is no learned-peer
  dialing;
- `max_num_inbound_peers = 0` and `max_num_outbound_peers = 0`;
- every existing validator and the one time-bounded joining node ID is listed
  in `unconditional_peer_ids` so only authenticated, explicitly listed inbound
  identities bypass the zero limit;
- the same IDs are `private_peer_ids` so addresses are not gossiped;
- `provider_node_ids` must exactly equal `validator_node_ids`: v1 does not
  authorize a privileged provider subset of the existing validator set;
- persistent peers contain only authorized validators/providers, and a receiver
  has every existing validator/provider as a persistent peer;
- the join approval binds chain ID, joining Comet node ID, validator public key,
  app-v20 compatibility, expiry, and snapshot height floor.

The in-process `sage-gui` state-sync path pins CometBFT v0.38.23. In that
version, the switch checks the authenticated node ID against
`unconditional_peer_ids` before applying the
inbound limit, and unconditional peers are excluded from normal inbound and
outbound counts (`p2p/switch.go` in the pinned module). SAGE does not rely on an
operator editing the generated Comet file correctly: it applies and validates
the effective profile locally before constructing/starting the node. The exact
unconditional and private sets must match `authorized_peer_ids`; every
`quorum.peers` persistent address must authenticate as a member of that set.

The root module also replaces upstream v0.38.23 with the provenance-recorded
local source subset in `third_party/cometbft`. It pins upstream source commit
`feb2aea4dc271d612129afc958cb844713ec792b`. That tag still declares core
semver `0.38.22`, so the standalone build stamps the commit and must report
`0.38.22+feb2aea4dc271d612129afc958cb844713ec792b`. Six coordinated overlays
are carried and copied into the standalone validator image:

1. `statesync/reactor.go` checks `syncer` under the reactor lock before
   rejecting an oversized advertisement received outside an active sync, so it
   cannot dereference nil;
2. `blocksync/reactor.go` recognizes only SAGE's direct or wrapped seal-abort
   sentinel as a graceful block-sync exit while shutdown is already underway;
3. `node/node.go` reads and retains that height bridge across consecutive
   empty-blockstore restarts, supplies it to the block-sync reactor, and ignores
   it only after a block is materialized;
4. `node/setup.go` enforces `commit -> state -> block-sync switch -> completion
   marker` and owns exact-only recovery of the sole configured-genesis cache
   written before asynchronous state sync;
5. `state/store.go` persists positive state and its effective-height bridge in
   the same synchronous bootstrap batch; and
6. `store/store.go` makes commit and marker writes synchronous and supports raw-
   database exact lone-commit recovery only after SAGE proves Comet state is
   empty. Raw inspection preserves malformed `blockStore` metadata instead of
   constructing a `BlockStore` that would panic while decoding it.

`make test-cometbft-patch` race-tests the inactive-syncer path, graceful-abort
classification, effective-height recovery across two consecutive empty-store
restarts, atomic state/height-marker bootstrap, every ordered bootstrap
boundary, marker encoding, exact genesis-cache recovery, and non-mutating raw
commit inspection plus exact commit-only recovery. The SAGE runtime tests hold
a post-switch consensus block call until seal while query/mempool calls fail
fast. The upstream module checksums, exact commit/runtime provenance, files,
license/NOTICE, and update procedure are recorded in
`third_party/cometbft/README-SAGE.md`.

The existing LAN non-validator join flow is not silently converted to this
profile: today the host does not add the guest node ID to its own allowlist.
The internet join ceremony must update both sides atomically or roll both back.

The receiver also checks every `ApplySnapshotChunk.sender` against the approved
provider set and returns it in `reject_senders` on mismatch. This is defence in
depth; it does not replace connection-level serving authorization.

Provider expiry is rechecked on every `ListSnapshots` and `LoadSnapshotChunk`;
an expired provider returns no advertisement or bytes. Receiver expiry is
stronger: it is checked at offer, on every chunk, before and after assembly and
isolated preparation, before bundle activation, and before every durable
journal/rename boundary. Assembly/preparation receives a context bounded by the
ticket deadline. Once any receiver check observes expiry, the controller
latches it permanently, so moving the clock backward cannot re-enable the
session. Expiry still does **not** cause CometBFT to disconnect an already-
authenticated unconditional peer: that ID remains in the process's in-memory
P2P profile. After the one-shot join, operators must disable the state-sync
role, replace the ceremony profile with the intended steady-state validator
mesh, and restart affected processes. Remove the joining ID if admission was
abandoned; retain it under normal quorum configuration if it became a
validator. After an uninterrupted success, a receiver must disable `receiving`
before its next restart because the one-shot role requires pristine local
application and Comet state. If startup instead recovers a matching pending
activation journal after a crash, it disables that recovered receiver role in
memory and resumes as an ordinary synchronized node for that process. This does
not rewrite the persistent configuration; the operator must still disable its
`receiving` field before a later restart.

The current validator transport is raw CometBFT TCP, not SAGE's independent-
chain libp2p federation relay. The joining node must reach each provider's P2P
listener (normally TCP 26656) for state-sync chunks and consensus traffic. It
must also reach at least two distinct configured HTTP(S) Comet RPC origins
(commonly 26657 or a TLS reverse proxy) for light-client verification. Those
origins allow no credentials, path, query, or fragment in SAGE configuration;
operators should use independently operated origins where possible.

Internet validators therefore still need mutually routable TCP addresses,
explicit port forwarding, or an operator-managed VPN. The federation relay
does not carry validator traffic. Supporting nodes without routable TCP/VPN is
a future consensus tunnel layer and is not delivered or implied by v11.9. The
same transport gap is separate from the still-missing integrated multi-process
release proof: unit/in-process success does not demonstrate a real provider and
pristine receiver exchanging the snapshot over authenticated Comet P2P.

## 7. Readiness and serving

`/ready` remains false from discovery through activation and projection rebuild.
After `sealed`, startup runs canonical scoped projection recovery against the
final bundle. Readiness becomes true only when:

- CometBFT has persisted and verified the synchronized height/AppHash;
- canonical scoped-state verification still succeeds on the writable store;
- every scoped content envelope has a matching rebuilt serving record and
  classification;
- the vault is unlocked when encrypted projection writes require it;
- local snapshot scheduling was rebound to the final Badger handle.

Consensus may continue while an encrypted projection waits for unlock, but
REST/MCP memory serving stays fail closed, matching the current readiness model
(`cmd/sage-gui/node.go:608-642`).

## 8. Operator configuration and endpoint behavior

The feature is nested under `quorum.state_sync`; `quorum.enabled` must also be
true. `serving` and `receiving` are mutually exclusive boot roles. Omitting
both preserves the default fail-closed endpoint behavior.

| Field | Role and validation |
| --- | --- |
| `serving` | Provider opt-in. Rejects receiver light-client fields. |
| `receiving` | One-shot receiver opt-in; requires the full pristine-node gate described below. |
| `snapshot_dir` | Optional. Defaults below `data_dir` to `state-sync/snapshots` for serving or `state-sync/receiving` for receiving. It must be a real, non-group/world-writable directory outside live `badger/`. |
| `authorization_file` | Required local JSON trust root. Relative paths resolve below `data_dir`. |
| `authorized_peer_ids` | Required canonical lowercase 40-hex Comet node IDs. At endpoint arming, the set must exactly equal the authorization's existing validators plus joining node. |
| `rpc_servers` | Receiver only: at least two distinct HTTP(S) origins, with no credentials, non-root path, query, or fragment. |
| `trust_height` | Receiver only: positive Comet light-client trust height. |
| `trust_hash` | Receiver only: 32-byte block hash in canonical lowercase hex. |
| `trust_period` | Receiver only: positive Go duration compatible with the chain's trust assumptions. |
| `startup_timeout` | Positive Go duration bounding discovery, assembly, Comet persistence, and sealing. Default `30m`; 15–30 minutes is the normal operator range. |
| `chunk_size` | Provider export chunk size, 64 KiB–8 MiB; default 4 MiB. |

Provider boot additionally requires pruning to be disabled: use top-level
`retain_blocks: 0` in the quorum configuration (the quorum default), and keep
the effective retention value at zero. A positive rolling window is rejected
before P2P starts because v1 rolling snapshots are not yet block-base aware and
an H+2-eligible receiver may need historical light/catch-up blocks. This is a
provider requirement; it does not change the receiver's local retention policy.

The operator installs the same ceremony data as strict JSON on the provider(s)
and receiver. `validator_public_key` is standard JSON base64 for the raw
32-byte Ed25519 key; `expires_at` is RFC 3339. The values below illustrate the
schema only and must be replaced with the ceremony's live values:

```json
{
  "chain_id": "sage-main",
  "joining_node_id": "0123456789abcdef0123456789abcdef01234567",
  "validator_public_key": "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=",
  "app_version": 20,
  "expires_at": "2026-07-16T12:00:00Z",
  "snapshot_height_floor": 125000,
  "validator_node_ids": [
    "89abcdef0123456789abcdef0123456789abcdef"
  ],
  "provider_node_ids": [
    "89abcdef0123456789abcdef0123456789abcdef"
  ]
}
```

This is deliberately not an invented unsigned network ticket. Local file
installation is the trust decision. The loader accepts at most 64 KiB and
rejects symlinks, non-regular or group/world-writable files, open-time file
replacement, unknown fields, trailing JSON, malformed/duplicate IDs, a wrong
chain/app version, expired approval, missing provider/validator sets, or a
provider set that differs in any way from the existing validator set. Use mode
`0600`. The receiver also proves that its live Comet node ID and local Ed25519
validator public key match the authorization. Chain IDs are lowercase
alphanumeric plus `.`, `_`, or
`-` (maximum 128 bytes and not `.`/`..`); node IDs are unique lowercase 40-hex,
the joiner is distinct from existing validators, and the non-empty provider set
must exactly equal those validators. Expiry must be in the future, and the
height floor must be positive. The exact equality deliberately prevents an
authorization file from creating a preferred provider subset.

The receiver is pristine only when all of these checks pass before Comet starts:

- `Info` reports height zero and an empty AppHash, and the raw Badger keyspace
  has no keys at all;
- every application-owned SQLite table is empty except `domains`, which must
  contain exactly the canonical schema seed rows and decay rates; FTS shadow
  storage does not excuse a populated `memories_fts` virtual table;
- Comet's persisted state is empty; startup may then remove only an exact lone
  valid seen-commit residue, after which each `state`, `blockstore`, `evidence`,
  and `tx_index` database must have no entries;
- the consensus WAL is absent or a zero-length regular non-symlink file;
- FilePV signing height/round/step are zero and signature/sign-bytes are empty;
  and
- genesis is not the single-local-validator shape that would make Comet skip
  state sync and initialize a new local chain.

Any stale memory, identity, credential, federation/policy row, modified domain
seed, Comet key, WAL byte, or signing record refuses the receiver role. This is
stricter than checking only application height.

Provider startup requires a committed positive-height app-v20 state at or above
the ticket floor and effective `retain_blocks=0`; a positive pruning window is
rejected until rolling snapshots are block-base aware. Before the Comet node
exposes P2P, SAGE maintains the provider root, writes the bounded canonical
latest-visible stream at the current committed height, restores it into an
isolated Badger database, and atomically publishes it only after app-v20
height/epoch/AppHash/scoped-state verification.
An existing exact height/AppHash catalog entry is reused after maintenance; a
same-height conflict fails startup. This is a startup publication, not the
operator-local rollback scheduler.

Provider maintenance removes only structurally proven SAGE-owned staging
leftovers and retains the newest eight valid owned snapshots. Export prunes to
seven before publication, then returns the set to eight. The newest two may be
waiting for their H+1/H+2 light blocks, leaving six deterministic older fallback
points; unrelated, malformed, or symlinked entries are preserved rather than
deleted. The temporary canonical source and chunk set are each bounded by the
4 GiB v1 maximum.

Endpoint behavior is then:

- `ListSnapshots`: return only verified public catalog entries at or above the
  authorization floor and no higher than `live_height - 2`; one canonical
  entry per height. A snapshot exported at `H` is therefore invisible until
  the chain has committed both `H+1` and `H+2`, which Comet's light client needs
  to verify a non-tip snapshot.
- `LoadSnapshotChunk`: select through that same catalog and rehash the chunk
  immediately before returning it. It never reads rollback paths.
- `OfferSnapshot`: accept only the one boot session after format, trusted
  AppHash, metadata, floor, authorization, and staging-disk checks. The staging
  filesystem must have at least `2 * snapshot_bytes + 256 MiB` available for
  verified chunks plus the assembled canonical stream. Failure is a terminal
  local boot-capacity error, not a provider rejection.
- `ApplySnapshotChunk`: enforce approved sender, exact session, bounds,
  out-of-order/idempotent duplicate semantics, per-chunk/full-stream hashes, and
  synchronous finalization before returning final `ACCEPT`. Before restoring a
  candidate, the data filesystem must separately have at least
  `4 * snapshot_bytes + 256 MiB` available for the restored Badger candidate,
  LSM/WAL expansion, and write/compaction overlap. A capacity failure is
  terminal and does not consume the eight-provider semantic fallback budget.
  Native preparation failures such as Unix `ENOSPC`/`EDQUOT` and Windows disk-
  full/quota equivalents are classified the same way even if the preflight had
  just passed. These are fail-fast capacity checks, not disk reservations.

The receiver finalizer first restores into a fresh isolated directory and
verifies exact height, epoch, AppHash, active app-v20 and canonical scoped state
while the old application remains open. A provider candidate that fails this
preverification is rejected without any journal or live rename, allowing
bounded fallback to another authorized snapshot. Only a successfully prepared
candidate enters the exclusive runtime lease: startup closes the old consensus
handle, rechecks/latches authorization across the journal and directory switch,
reopens the verified database without migrations, constructs a complete
replacement `SageApp` bundle, and publishes it as `activated_pending_comet`.
From the `prepared` transition onward, any local close, journal, rename, reopen,
or bundle-activation failure is terminal and never consumes semantic provider-
fallback budget.

The main startup goroutine then waits for Comet's asynchronous persistence and
post-switch marker. Consensus block calls started after Comet's switch wait
behind the seal, while externally triggerable `Query` and `CheckTx` fail fast
throughout every unsealed phase, including PendingComet. Startup then seals and
releases pending consensus block execution, freezes the bundle, and only
afterward constructs projection, snapshot, search, REST, dashboard, MCP,
federation, voter, and other workers. Ordinary consensus ABCI methods remain
blocked until that seal.

Any missing authorization, provider/validator set mismatch, effective P2P
drift, non-pristine receiver, unexpected second offer, exhausted bad-candidate
fallback, insufficient disk preflight, timeout, local snapshot metadata, or
incomplete crash recovery fails closed. After success, disable `receiving`
in persistent configuration before restart, including after a recovered role
was disabled only in memory. A provider should disable `serving` after the
ceremony; an expired file makes a new armed startup fail, while an already
running process stops serving chunks but retains its existing P2P connection until
reconfiguration/restart.

## 9. Remaining release gates

The unit/integration surface now covers strict authorization, effective P2P
hardening, endpoint/session failure cases, exact bundle replacement, journal
recovery, serving-order admission, and provider export/receiver finalization.
CI also runs the local six-overlay Comet regression gate, including both
reactors, two-restart effective-height durability, atomic bootstrap-marker
persistence, and the state-sync reactor under the race detector.
The app-v20 subprocess fault oracle uses independent process stores to exercise
SIGKILL/replay, partition/catch-up, roster revision, and pinned old/new ballot
denominators.

The real four-validator Docker harness exercises complete Comet/ABCI process
SIGKILL and firewall-based one-validator and 2/4 Comet P2P partitions. Stable
common-network peer IPs are rejected on P2P port `26656` in per-Comet
`iptables` chains while each validator's private ABCI network remains intact;
nonzero rule counters prove the partition was exercised. Test-only,
runtime-verified `tcp_retries2=3` bounds stale established-socket eviction on
Docker Desktop kernels without `INET_DIAG_DESTROY`. Host REST and Comet RPC
ports are loopback-only, and every run uses an owned temporary genesis fixture.
The harness proves the expected peer split and zero height movement on both 2/4
halves, heals the firewall rules, then requires equal latest Comet block hash
plus matching live ABCI height/AppHash and `catching_up=false` on all
validators. It does **not** yet
drive an authorized provider-to-pristine-receiver state-sync transfer or real
signed app-v20 scope formation/revision. Its standalone Comet image builds the
exact v0.38.23 source commit and reports
`0.38.22+feb2aea4dc271d612129afc958cb844713ec792b`; it copies the same six
overlays as `sage-gui`. A `v11.9*` release enables both
`V119_REQUIRE_SCOPED_RECONFIG=1` and
`V119_REQUIRE_AUTHORIZED_STATE_SYNC=1`, so topology success cannot be counted
as either missing proof.

Before v11.9 can be called release-ready, an integrated-node Docker path must
run real `sage-gui serve` processes and combine the endpoints with unauthorized-
peer rejection, provider/receiver restart recovery, post-sync catch-up, and
signed scope reconfiguration, then pass full Go/race/lint, SDK/frontend, and
adversarial review gates. The existing split `amid` plus standalone Comet image
passes raw `SageApp` to ABCI and therefore cannot count as this proof: it does
not own the boot runtime, whole-bundle activation, or seal-before-serving
boundary.
