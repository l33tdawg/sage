# v11.9 Multi-Process Fault Harness

**Status:** executable composite crash/partition and signed scoped-
reconfiguration gate delivered. Authorized network state sync is owned by its
separate integrated `sage-gui` wire gate; that gate's final exact-source cold
pass remains pending and is not credited by this document.

This fault harness has two intentionally separate, complementary tiers. The
Docker tier proves real CometBFT TCP partitions. The deterministic subprocess
tier proves the signed app-v20 state-machine ceremony over independent process
stores. A held subprocess is never described as a TCP partition.

## Real CometBFT/ABCI process gate

Run:

```bash
make v119-chaos
```

`deploy/scripts/run-v11.9-chaos.sh` creates an isolated four-process Docker
Compose project (`sage-v119-chaos`) and builds the current source tree by
default. Three process identities begin in genesis; node3 is a live full node
with its own Comet/ABCI pair but no voting power. The gate then performs bounded
assertions over the real processes. The owned pre-start fixture normalizes each
genesis validator to `P=100`, independent of CometBFT generator defaults, so
the power revision is the exact integer change `100 -> 120` and the later
partition geometry is stable across environments:

Before any process starts, the gate parses all four generated Comet
configurations and requires `[mempool] recheck = true`,
`max_tx_bytes = 1048576`, and an empty `wal_dir`. The first two settings are the
supported app-v20 external-Comet profile; the empty WAL makes each crash/restart
assertion exercise persisted application and chain state rather than replaying
an uncommitted local mempool. App-v20 still seats bounded stale resource-invalid
entries and drains them with deterministic FinalizeBlock Code 111, so liveness
does not rely on recheck being the only eviction mechanism.

1. Generate a fresh per-run Ed25519 operator, derive its identity instead of
   hard-coding trust, register it as the pre-app-v9 chain admin, and drive the
   three validators through the real signed upgrade ladder to app-v20. Each
   ABCI gateway retains its own mounted Comet private-validator key, and the
   three distinct live voters authorize every quorum-gated upgrade.
2. Govern an add of live node3, a bounded node3 power increase, and removal of
   node2. For each operation, query the exact old set at execution `H+1` and the
   exact new ID/power set at `H+2` from every Comet RPC. SIGKILL and restart the
   affected complete Comet+ABCI pair after each change, then re-query the
   historical `H+2`, the latest effective set, and all four authenticated ABCI
   contexts. The latter must expose the exact persisted Badger roster; after
   removal node2 reports `validator_active=false`, remains absent from sampled
   commit signatures, and its restarted gateway's attempted governance vote is
   rejected by consensus.
3. With final voting powers `node0=P`, `node1=P`, `node3=P+d` and node2 at zero,
   firewall lower-power validator1 away from the other processes. Require the
   connected `/net_info` views to contain exactly nodes0/2/3 and the isolated
   view no peers, then use authenticated transactions to prove node0+node3 have
   strictly greater than two-thirds voting power and progress while node1
   falls behind. Healing must catch node1 up to the same block/AppHash.
4. Firewall processes 0/1 from 2/3 in both directions. The two halves carry
   `2P` and `P+d`, respectively, and the fixture bounds `d <= P/2`, so neither
   has greater than two-thirds of `3P+d`. After all four `/net_info` views show
   exactly their same-half peer, stabilize heights for 8 seconds and prove zero
   movement for at least 12 seconds. Healing must restore progress and converge
   all four processes on the same latest block hash, live ABCI height/AppHash,
   and `catching_up=false` status.

Partitions use a dedicated `SAGE_V119` `iptables` chain inside each Comet
container, keyed to stable common-network peer IPs, hooked into INPUT and
OUTPUT, and rejecting only TCP flows with P2P port `26656` at either endpoint.
The harness requires every chain's packet counter to become nonzero before it
accepts a peer split. Healing deletes the hooks and chains. Per-validator
private networks keep each Comet-to-ABCI link alive, so the test isolates
consensus transport rather than killing the application. P2P is not
host-published; REST/metrics and Comet RPC host ports bind only to `127.0.0.1`.

Each test-only Comet container also sets and runtime-verifies the network-
namespace-local `net.ipv4.tcp_retries2=3`. This bounds eviction of established
P2P sockets after the firewall is inserted. That is necessary on Docker
Desktop's LinuxKit kernel, where `CONFIG_INET_DIAG_DESTROY` is disabled and
`ss -K` silently leaves established sockets alive. The setting does not create
the partition: the port-scoped firewall rules do. It only prevents a rejected
accepted socket from remaining in Comet's `/net_info` registry for the Linux
default retransmission horizon.

Every run creates a canonicalized, marker-owned `mktemp` work directory holding
the genesis fixture plus bind-mounted Postgres and ABCI stores, and uses the
dedicated `sage-v119-chaos` Compose project. This keeps the fault gate isolated
from Docker's shared named-volume pool. Cleanup removes only that project and
owned work directory; failure to tear down, verify removal, or leave no project
containers is itself a red gate. It never invokes
`make down-clean`. Set
`V119_CHAOS_KEEP=1` only when inspecting a failed cluster manually. Set
`V119_CHAOS_REBUILD=0` only when deliberately reusing the named local images;
the release default rebuilds the current tree.

The standalone validator build pins v0.38.23 source commit
`feb2aea4dc271d612129afc958cb844713ec792b`. Because that source tag still
declares core semver `0.38.22`, the exact stamped runtime string required by the
harness is `0.38.22+feb2aea4dc271d612129afc958cb844713ec792b`.

`.github/workflows/v11.9-fault-gates.yml` runs both tiers with bounded job
timeouts and cached Docker builds. Normal CI calls that reusable workflow. All
release publishers depend on it; a `v11.9*` tag also sets the real scoped-
reconfiguration and authorized state-sync requirements. The scoped flag is
satisfied only by the composite proof described here, while the state-sync flag
is owned by the integrated wire gate.

The split `amid` fixture now reaches app-v20 and proves live validator-set
governance, but it still does not duplicate the scoped formation/revision and
pinned-ballot state-machine ceremony. The release proof composes it with the
race-enabled subprocess oracle below. In CI the scoped requirement is routed
to the prerequisite subprocess job; a direct
`V119_REQUIRE_SCOPED_RECONFIG=1` chaos invocation runs that companion oracle
after the Docker topology succeeds. `V119_REQUIRE_AUTHORIZED_STATE_SYNC=1`
independently invokes the integrated provider-to-receiver wire gate.

Only the explicitly selected `v119-state-sync-fixture` Docker target contains
the `v119testfixture` amid/sage-gui binaries and authenticated helper. The tag
shortens proposal cooldown and upgrade-delay timing and exposes the existing
test pause hook; it does not weaken authorization, identity, quorum, P2P,
state-sync, sealing, or serving rules. The default/final production target is
built from `production` and copies only `/app/amid`. Both fixture images receive
the same frozen `V119_STATE_SYNC_SOURCE_ID`. Its v4 tree encoding length-frames
every path and records object kind, mode, content length, and content digest, so
file bytes cannot manufacture an ambiguous record boundary. It walks the exact
deny-by-default Docker source model without consulting local or global Git
ignores, binds both fixture Dockerfiles plus the complete fault-gate control
plane, and rejects policy drift or per-Dockerfile ignore overrides. The harness
checks both image labels and the current source/control identity before and
after every composed gate.

## Deterministic application subprocess oracle

Run:

```bash
make v119-multiprocess-app
```

`internal/abci/v11_9_multiprocess_fault_test.go` launches four copies of the
test executable with independent Badger and SQLite databases. It provides a
fast deterministic proof of the production app-v20 state machine:

- a live child is withheld from an exact block interval and later replays each
  missed block with an intermediate AppHash comparison;
- a child is externally SIGKILLed after speculative `FinalizeBlock` execution
  but before the single atomic `Commit`; reopening proves that no partial
  write survived, then replays the same mixed state transition and AppHash;
- a distinct non-validator admin signs the app-v20 request authorization while
  outer validator keys form revision 1 and later approve revision 2; no scope
  record is fixture-seeded;
- a four-member ballot remains pinned to revision 1 after a governance-approved
  revision 2 reduces the scope to three members;
- three of four commits, while exactly two of three remains pending until the
  third vote.

This tier is a real OS-process/reopen proof, but its parent process is the block
delivery transport. It is not evidence for CometBFT TCP behavior, P2P
authorization, NAT traversal, endpoint serving, or network state-sync.

## Proof boundary

The scoped gate is intentionally a composition, not an integrated-topology
claim: the subprocess oracle proves exact dual-principal envelopes, prospective
roster change, pinned old/new denominators, crash replay, and held-replica
catch-up; the Docker oracle independently proves real Comet P2P partition,
strict 2+2 halt, healing, and block/AppHash convergence.

Authorized state sync remains a separate integrated `sage-gui serve` proof
because split `amid` + standalone Comet passes raw `SageApp` to ABCI and does
not own `BootStateSyncRuntime`, whole-bundle replacement, or seal-before-serving.
`make v119-state-sync` owns reciprocal unauthorized denial, provider/observer
light origins, the durable-prepublication receiver crash/restart path, the
separate non-crashing `session < seal < REST` receiver path, provider restart,
and post-sync projection/block/AppHash convergence. Its final exact-source cold
pass is still required. Provider configuration must have effective
`retain_blocks=0` until rolling snapshots are block-base aware. `Query` and
`CheckTx` must fail fast throughout every unsealed phase, including PendingComet;
only consensus block calls may wait behind the PendingComet seal. Service
admission follows seal. The shutdown path must prove that only the SAGE seal-
abort sentinel exits block sync gracefully. Raw malformed block-store metadata
must be preserved/rejected without a constructor panic. Preflight and native
disk-full/quota failures must remain terminal rather than consuming provider-
fallback budget; local activation failures from `prepared` onward are terminal
under the same rule.

This release proof is distinct from Internet reachability. Validators still
need mutually routable Comet TCP, explicit port forwarding, or an operator VPN,
plus reachable Comet RPC origins. Federation is not a validator tunnel. A
future tunnel layer is still required for nodes without that connectivity and
does not remove the pending integrated real-transfer release gate.
