# v11.12 clean-machine acceptance

Date: 2026-07-23

Status: **earlier structured proxy evidence retained; federation end-to-end closure is open pending the real two-node gates below**

This is the release team's substitute for the planned 3–5-person nontechnical
usability study. It is useful release evidence, but it is not evidence that real
first-time users understood the product. We must keep that distinction visible.

## How to run the proxy fairly

Use a disposable SAGE home on a clean browser profile. During each scenario, the
operator may use only the visible product. No source, terminal, API, docs, or hints.
An observer may record actions and ask only:

1. What do you think this choice will do?
2. What do you think remains private or safe?
3. What would you do next?

Record a failure if the operator needs hidden knowledge, cannot predict a privacy
or destructive action, cannot recover using the visible UI, or chooses a dangerous
action while believing it is safe. Source inspection and automated tests can
confirm implementation after the attempt; they cannot rescue a failed attempt.

## Findings fixed during the run

- Replaced onboarding implementation terms such as “semantic embedder,” “Ollama,”
  “wire an AI tool,” and “agent identity” with task-oriented language.
- Added an explicit **Finish setup** path when sharing was selected, so sharing
  remains optional rather than a trap.
- Gave passphrase and network pairing-code fields meaningful accessible names.
- Hid encryption algorithm, key-derivation, and vault-path details behind a
  collapsed **Technical details** disclosure.
- Updated the written first-run guide and roadmap summary from the obsolete
  three-step wizard to the current create/join, privacy, and recovery flow.
- Serialized multi-member group invitations and limited the member picker to
  active trust generations, so one submission cannot race the signed roster or
  reuse a revoked/re-paired historical identity. SAGE verifies and adds each
  selected trusted SAGE automatically; brief idempotent bootstrap retries stay
  inside the create flow and a final partial result remains visible.
- Routed member-signed self-role changes through the group owner for roster-head
  allocation and countersigning. Concurrent guests retain control of their own
  consent while the owner prevents two valid changes from occupying the same
  journal sequence.
- Made **Refresh** run and await a bounded signed-journal reconciliation pass
  before reloading the local projection; it now shows **Refreshing…** instead of
  behaving like a stale local-only reload.
- Bumped the CEREBRUM asset version so the fixed JavaScript is not masked by an
  older browser cache during upgrade or acceptance.
- Same-network join now binds the ceremony to the exact SAGE executable, including
  development builds, and discards only old-chain projection receipts while
  preserving the guest's local memories before canonical replay.
- Recovery now verifies that a correctly shaped recovery key belongs to the
  existing vault before any backup or rewrite, then establishes the normal
  CEREBRUM session after a successful passphrase reset.
- Labeled the memory-search filters, cleanup and preference switches, cleanup
  sliders, recall sliders, reranker switch, and per-domain access switches so
  screen readers announce the action rather than only the control type.

### Route and failure-state evidence

The connection row and ceremony must preserve these distinct states. Behavioral
render-state tests prove precedence and copy; the physical-node route gate above
alone proves real route selection and recovery.

| State | Required visible outcome | Required trigger |
| --- | --- | --- |
| Direct | **Direct**, with no relay warning | Peers reachable on LAN/direct P2P |
| Secure relay | **Secure relay**, explaining end-to-end encryption | Direct path unavailable, relay reservation/dial succeeds |
| Degraded | **Degraded**, with the backend reason | Preferred route fails while a fallback still carries traffic |
| Offline | **Offline** and retry | Both prepared routes unavailable/time out |
| SAGE locked | **SAGE locked** and unlock guidance | Local vault prevents the requested operation |
| Older SAGE | Compatibility through Direct; no promise of roaming | Peer lacks current route diagnostics/capability |
| Route failed | **Route failed**, not generic unreachable | Route preparation/dial fails without a security mismatch |
| Trust check failed | Pair-again guidance | Agreement missing, expired, or revoked |
| Security blocked | Explicit certificate/pin/identity mismatch and “sent no data” | Fail-closed authenticated dial with a mismatched trust proof |

### Copy deletion invariant

Pause, permission removal, group removal, and trust revocation stop future access
or synchronization. They do **not** remotely erase memories already copied onto
another sovereign SAGE. Acceptance must confirm the warning before revocation and
must delete retained local memories separately when erasure is desired.

## Release decision

The earlier structured proxy gate is green only for the product paths it directly
exercised. It does not close the federation gates above. One
protected-artifact partial remains: the acceptance DMG is Developer-ID signed,
mounts, copies with its signature intact, and its installed-copy node launches on
isolated ports, but local credentials cannot notarize or staple it. Gatekeeper
correctly reports `Unnotarized Developer ID`, so the protected release workflow
must produce the notarized/stapled artifact for a true clean-machine Finder/Launchpad
open. The release owner accepts this structured expert proxy for v11.12 because
independent nontechnical testers are unavailable; the real-user study remains a v12
criterion and the full spoken VoiceOver matrix remains in v11.14. This evidence must
still not be described as human usability testing. The federation lab used three
disposable loopback nodes and an acceptance-only three-block fork delay. The shorter
delay affected only the disposable acceptance environment; the lab otherwise
exercised the production federation paths, including the production fixes listed
above.
