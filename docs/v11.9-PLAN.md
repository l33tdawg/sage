# v11.9 — Domain-Scoped Quorum and Canonical Replica State

**Status: released in v11.9.0
(2026-07-17).** The app-v20 scoped-quorum model, canonical content,
projection recovery, guided operator surfaces, ordered replay, and local
snapshot recovery are implemented. The consensus-only `sage-gui` state-sync
path now also has strict local join authorization, a validator-only CometBFT
v0.38.23 peer profile, coordinated six-file Comet hardening, a
canonical latest-visible state stream, verified provider startup export with
H+2 eligibility and bounded retention, real requester-anonymous ABCI serving
endpoints, a full pristine/disk-gated one-shot receiver, isolated candidate
preverification with bounded provider fallback, latched authorization expiry,
crash-safe whole-bundle activation, a bounded Comet persistence-and-seal wait,
and ordinary-ABCI plus final-store serving barriers.

The same v11.9 release also completes the colleague-sharing authorization model
for independent-chain federation. JOIN is trust-only and installs an explicit
empty peer policy; each node later replaces its own existing-domain Read/Copy
snapshot, and Copy is effective only when the receiver independently subscribes.
Cross-host Write stays reserved and returns `501` until a connection-generation-
bound consensus ingress capability exists. Exact chain/operator/CA/epoch and
group-roster gates, plus a shared agreement-mutation boundary across REST and
dashboard paths, make permission changes, JOIN activation, narrowing, and
revocation fail closed under concurrency.

App-level multi-process SIGKILL/held-replica/reconfiguration coverage and a real
four-validator Comet/ABCI crash/partition harness now exist. The signed scope
gate composes a race-enabled dual-principal OS-process ceremony with the
independent real-Comet Docker partition proof; the held subprocess is not a TCP
partition claim. The split Docker topology still does not itself drive
authorized state sync; that proof has its own integrated `sage-gui serve`
harness. This document does not claim that v11.8 synchronization groups provide
BFT or that raw Internet validator reachability is solved. Internet validators still require routable TCP, port forwarding, or
an operator VPN today. A tunnel layer is separate future, out-of-scope work,
not a v11.9 release gate. The exact-source cold execution of
`make v119-state-sync` passed on source identity
`7535074403bda428337e20a1e5edf7cf7c6b9a554fda690cf837bd3748d5f53c`;
the complete CI/security matrix independently guards branch and tag publication.

## 1. Outcome and boundary

v11.9 makes a selected set of domains into canonical, recoverable replicated
state inside **one SAGE consensus chain**. The participating nodes are
validators of that chain. A scope has an on-chain roster, an explicit domain
allowlist, deterministic member weights, and a greater-than-two-thirds
application quorum for scoped memories. An offline validator can catch up by
ordered committed-block replay today. Canonical scoped content also survives a
verified local snapshot with a discarded SQL projection. The CometBFT network
state-sync implementation and integrated Docker harness are explicit and
opt-in, and the harness's final exact-tree cold pass is part of the permanent
release gate.

This is intentionally not an attempt to turn the v11.8 cross-chain
Synchronization Group overlay into cross-chain BFT:

- v11.8 groups connect independent chains. Their SQLite tables, controller
  journal, pairwise treaties, and locally signed imported copies remain an
  off-consensus sharing feature.
- A v11.9 quorum scope is contained within one chain. Its members are existing
  validators identified by their on-chain validator/agent IDs; it does not use
  `member_chain_id` as a consensus identity because every cluster member has the
  same chain ID.
- The v11.8 SQLite group model may remain a UI/projection and migration aid, but
  is never an authority for v11.9 admission, quorum, revocation, or recovery.
- Federation and Natter remain transport only. They neither participate in the
  scope quorum nor receive unselected-domain metadata.

That separation is the release's first safety property: no selected scope can
make a private domain, an unrelated federation peer, or a v11.8 controller
part of consensus authority.

It does not freeze the independent-chain federation product in its v11.8 form.
For fresh v11.9 links, the human-verified JOIN establishes identity trust only
and grants no domains. Each source can later grant Read and/or Copy over
concrete domains it controls; each receiver separately chooses whether to keep
offered copies. Live Read remains borrowed and off-consensus, accepted copies
enter the receiver through its ordinary locally governed transaction path, and
remote Write is unavailable. Direct sharing and synchronization-group relays
both remain subordinate to the exact active JOIN identity and generation.

## 2. Existing substrate and the design decision

The existing chain already gives v11.9 three load-bearing primitives:

1. CometBFT orders the transaction and replicates Badger state to the current
   validator set.
2. `TxTypeMemoryVote` and `validator.CheckQuorum` provide the application-level
   accept/reject stage after block inclusion.
3. Badger state is AppHash-covered and locally snapshotable; PostgreSQL/SQLite
   is a serving projection, not the consensus source of truth. The ABCI network
   state-sync methods are fail-closed by default and become reachable only for
   an explicitly authorized boot role. Their real Docker transfer remains a
   release proof, not an assumption.

The current memory model keeps only hash/status/classification in Badger and
keeps content and serving indexes in the projection. That is insufficient for a
new validator or a state-synced node to prove it holds the complete selected
domain. Therefore v11.9 adds a **canonical scoped-content mirror** in Badger.
For a scope-selected memory the delivered envelope contains id, content, type,
domain, classification, confidence, timestamps, author, and content hash. It is
written only by the deterministic app-v20 FinalizeBlock path and is covered by
AppHash. Above app-v20, the submitter signs one backward-compatible extension
containing at most 32 tags of 128 UTF-8 bytes each. The REST and federation
paths reject padded/invalid values, sort bytewise, and deduplicate before
signing. The extension remains dormant during pre-v20 decode/replay; after the
fork the app activates and validates it at the deterministic block boundary.
Scoped envelopes include the canonical list, so projection rebuild cannot
silently drop it. Untagged and every historical pre-v20 transaction retain
their prior bytes and behavior.

Every app-v20 block, including a mixed ordinary/governance block and the
validator update produced by a timer-driven governance transition, is evaluated
against one speculative Badger transaction. `Commit` adds the height, AppHash,
and handshake tuple to that same transaction and publishes the cloned in-memory
state only after the database commit and sync succeed. A crash or failure before
that point leaves no proof claim, nonce, proposal, validator-set change, or
ordinary handler write behind, so the complete block can be replayed from the
last committed height. No app-local FinalizeBlock result journal or
governance-only block isolation is required: after a successful app `Commit`,
CometBFT's persisted response/handshake recovery owns the
app-height/store-height reconciliation.

The atomic ceremony is deliberately bounded: at most 64 transactions and 1 MiB
of raw transaction bytes per block; 512-byte identifiers, 64 KiB metadata and
encoded-record ceilings, 512 KiB content, 64 collection entries, and 100
validators. An authenticated, registered administrator's target-20 proposal
must be alone in its bootstrap block. Only after those checks does it perform
the exhaustive legacy-state resource audit and create the governance proposal;
the audit-complete marker is written in the same speculative Badger transaction
and becomes authoritative only at `Commit`. A raw, forged, rejected, or mixed-
block target-20 transaction changes neither legacy admission semantics nor the
marker. Cancellation, rejection, or expiry clears the marker, so a later retry
must audit again. Once the marker is committed, readiness uses bounded checks
of the current validator roster, active proposal, and pending plan instead of
re-scanning historical keyspaces on every block. Epoch processing likewise
reads statistics only for the bounded current roster, leaving stale removed-
validator evidence available for repair.

The v11.9 binary applies those static field and one-transaction raw ceilings in
advisory `CheckTx` even before activation, while pre-v20 `ProcessProposal`,
`FinalizeBlock`, TxResults, nonces, and AppHash retain legacy behavior. After
the marker commits, `PrepareProposal` seats a bounded resource-invalid entry
rather than omitting it forever; `FinalizeBlock` returns deterministic code 111
before mutation and Comet removes the included entry at commit. An authenticated
bootstrap that cannot fit `MaxTxBytes` is omitted without being mixed, while
ordinary transactions that fit are still proposed. These rules remove a hidden
liveness dependency on external Comet rechecking without weakening the global
64-entry/1-MiB budget.

The legacy audit applies the same bounds to existing records and rejects
oversized persisted validator or PoE-weight keyspaces before
`ReplaceValidators`/epoch cleanup can amplify them into an oversized Badger
transaction. Current validator IDs must be canonical lowercase Ed25519 public-
key hex and bind to the hydrated in-memory public key. During the committed
ceremony, governance/store read failures, failed proposal application, epoch
persistence failures, and activation audit failures abort and discard the whole
speculative block instead of publishing a partial success.

The selected implementation model is therefore:

```
agent submit → CometBFT block → app-v20 scope lookup
                                    ├─ unscoped: current memory lifecycle
                                    └─ scoped: proposed + canonical content mirror
                                                     ↓
                    eligible scoped validators cast normal memory votes
                                                     ↓
                        >2/3 scoped weight → committed canonical replica
                                                     ↓
       ordered replay / verified local or network snapshot → rebuild SQL/serving indexes
```

The normal global validator quorum remains unchanged for unscoped domains. A
scope does not alter CometBFT's block-finality threshold; it controls the
SAGE-level acceptance of memories in its explicit domain allowlist.

## 3. Invariants

Any implementation or optimization that violates one of these is a no-go.

1. **Pre-app-v20 replay identity.** With no applied tagged `app-v20` record, the
   new binary must reproduce historical TxResults and AppHashes byte-for-byte.
   There must be no new consensus state write, `ProcessProposal`/`FinalizeBlock`
   acceptance branch, validator-side request construction change, or changed
   `Info().AppVersion` on that path. Forward-compatible advisory `CheckTx`
   resource hygiene is allowed because it does not execute during block replay.
2. **One-chain authority.** A scope member is an active on-chain validator of
   this chain. Federation identities, SQLite rows, HTTP headers, and controller
   metadata are never evidence of consensus membership.
3. **Explicit domain isolation.** Only an exact selected domain or its declared
   subtree belongs to a scope. An unmatched domain uses the existing global
   quorum. Scope lookup must be deterministic, reject overlapping active scopes,
   and disclose no unselected-domain roster or content metadata.
4. **On-chain source of truth.** Scope roster, revision, active/retired state,
   selected domains, and canonical scoped records live in Badger and are
   AppHash-covered. PostgreSQL/SQLite caches may be discarded and rebuilt.
5. **Pinned denominator.** A memory's quorum denominator is the active scope
   roster and its exact integer weights as of the submit block. Later member
   changes cannot change an already-proposed memory's denominator.
6. **Strict supermajority.** Acceptance requires `accept_weight * 3 >
   total_weight * 2`, not a floating-point approximation. A three equal-weight
   member scope with only two online members does not continue if the chain's
   own configured BFT rules require more than two thirds.
7. **Removal and revocation fail closed.** A roster change only affects future
   proposals after its activation block. It cannot resurrect a revoked member's
   vote, widen an existing scope, or remove the pinned evidence needed to finish
   an older proposal. New scope writes pause if the canonical roster is absent,
   malformed, retired, or inconsistent with the active validator set.
8. **Canonical content recovery.** A scoped memory cannot be reported healthy
   until its canonical envelope is present and hashes to the normal memory hash.
   Local snapshot restore, projection rebuild, provider export, isolated
   receiver preparation, and promoted-directory verification all apply that
   same check. The opt-in endpoint path and integrated real-network harness are
   implemented; the final cold execution still gates release.
9. **No implicit privilege.** Scope membership gives only scoped validator vote
   weight. It grants neither domain ownership, agent RBAC, federation access,
   administrator authority, nor permission to read unselected records.
10. **Determinism.** App-v20 state codecs use fixed-width integers or
    length-prefixed byte strings, sorted collections, a leading schema-version
    byte, bounded decode sizes, and no maps/time.Now/external I/O in
    FinalizeBlock. Floating-point fields inherited from an already accepted
    transaction are encoded as raw fixed-width IEEE-754 bits and are never used
    for quorum arithmetic.

## 4. App-v20 activation

v11.9 consumes the next unused application version: `app-v20`. It must use the
same governed upgrade machinery as current post-app-v8 upgrades:

- canonical name `app-v20`, target version `20`;
- exact predecessor `app-v19`: target 20 is not a skip-ahead upgrade and is
  rejected from app-v1, app-v7, or any other committed version;
- existing upgrade-propose / validator-vote path, so a >2/3 validator-power
  governance decision schedules the fork. Target 20 is forced through this path
  even if an imported chain has inconsistent lower-gate bookkeeping; it can never
  fall through to the legacy single-signer plan path;
- an exact chain authorization commitment. The runtime derives
  `hex(SHA-256("sage/governance-delegation-domain/v20\x00" || chain_id))`
  from the non-empty, at-most-50-byte CometBFT chain ID, without trimming or
  case folding. `UpgradePropose.governance_domain` is part of the signed wire
  payload, governance proposal payload, and pending plan approved by quorum;
- the existing 200-block minimum activation floor; and
- strict consensus activation at `H_activation + 1`.

The v11.9 ceremony discriminator is the exact triple `name == "app-v20"`,
`target_app_version == 20`, and a canonical non-empty 32-byte lowercase-hex
`governance_domain`. Numeric target 20 by itself is not the tag: v11.8 accepted
arbitrary future upgrade targets, so an empty domain and every malformed or
non-canonical historical trailing-domain value retain the legacy decoder,
proposal JSON (`omitempty`), scheduling, activation, and halt behavior forever.
They do not select atomic finalization, freeze validator reconfiguration,
materialize the delegation domain, or arm the live app-v20 gate.

Before anyone broadcasts the exactly tagged target-20 transaction, **every
participating validator, not merely a >2/3 voting quorum, must be running the
exact frozen v11.9 source/digest**. Older binaries do not understand the signed
domain tail; a mixed-binary ceremony can therefore interpret different signed
payloads and is outside the supported rolling-upgrade boundary. Upgrade all
validators first, verify the identical artifact, and only then begin the
governance ceremony.

For SAGE-owned Comet runtimes, the mempool is volatile and the effective v11.9
profile uses `recheck=true` with a 1-MiB `max_tx_bytes` ceiling. An
operator-managed external Comet in `amid` socket mode cannot be verified through
standard RPC: configure the same ceiling, keep rechecking enabled as hygiene,
and restart the Comet process before the ceremony if it may retain pre-rollout
mempool entries. The deterministic Code-111 drain prevents bounded stale entries
from blocking progress even when recheck is disabled; the restart/cap rule
excludes an individually oversized entry that cannot fit the application block
budget at all.

The implementation adds a dormant `appV20AppliedHeight`, refresh-on-boot arm,
`postAppV20Fork(height)`, `currentAppVersion()` top case, and
`MaxSupportedAppVersion() == 20` in one change. It must also extend every lower
**additive** post-activation helper that app-v20 subsumes, while deliberately not
turning on the independent app-v18 administrator override. The test suite must
exercise skip-ahead rejection, activation-boundary replay, constructor recovery,
and the dormant AppHash equality proof.

App-v20 is the first behavioral change in this release. It does not alter any
pre-v20 transaction encoding; instead it enables the new scope transaction
types and state writes only at heights strictly after activation.

At the activation block, consensus validates and writes the 32-byte domain to
`state:governance_delegation_domain_v20` *before* `MarkUpgradeApplied` consumes
the pending plan and retains the exact proposal-time legacy-resource-audit
marker. Crash replay with an already-consumed tagged plan requires both keys;
reopening a tagged app-v20 database or importing a network state-sync image
fails closed when the domain is malformed or the marker is absent/malformed.
An old applied target-20 record with no domain remains a dormant legacy record
and never arms app-v20. Requiring the two-key tag prevents a stray domain key
from reclassifying it. The off-consensus upgrade auto-voter independently
derives the value from its actual CometBFT chain ID and refuses to accept a
target-20 proposal with a missing or different commitment.

Activation also binds the application roster to CometBFT's exact prior
validator set. `DecidedLastCommit` must contain exactly one vote entry for every
current SAGE validator, with the Ed25519-derived Comet address and identical
power. Empty commits, duplicate addresses, missing or extra validators, and
power/key mismatches refuse activation before the plan is consumed. Validator
reconfiguration remains frozen while the target-20 plan is pending, preventing
the approved activation roster from drifting underneath that check.

## 5. Canonical state model and codec

All v11.9 keys are in Badger and use a new `state:scope:` namespace. The key
layout is fixed before code is merged:

| Key | Value | Purpose |
| --- | --- | --- |
| `state:scope:<id>` | `ScopeRecordV1` | roster head, revision, state, domains, members |
| `state:scope-domain:<domain>` | scope id | exact deterministic domain lookup |
| `state:scope-proposal:<memory-id>` | `ScopeBallotV1` | pinned roster/weights and vote lifecycle |
| `state:scope-content:<memory-id>` | `ScopedMemoryV1` | canonical recoverable submitted envelope |
| `state:scope-revision:<id>:<n>` | SHA-256 head | immutable audit/revision anchor |

Each value begins with `0x01` (schema version), followed by a bounded,
length-prefixed binary encoding. Strings and byte slices are prefixed by a
big-endian uint32 length; member and domain vectors have a uint32 count; ids,
members, and domains are sorted bytewise before encoding. `uint64` is used for
revision and weight. Empty or duplicate fields, unknown enum values, malformed
UTF-8 domain strings, oversized fields, and trailing bytes are decode failures.

`ScopeRecordV1` contains:

```
schema_version | scope_id | revision | state(active|paused|retired)
controller_validator_id | created_height | updated_height
domains[] { domain, subtree }
members[] { validator_id, assigned_weight, joined_revision, active }
```

`assigned_weight` is an integer, non-zero, and is distinct from the current
float PoE reputation score. In schema v1 it is the complete governance-assigned
allocation: `ScopeBallotV1.effective_weight` copies it verbatim at submission.
Live PoE does not reweight a scoped ballot; ordinary unscoped quorum continues
to use the existing PoE path. Any later formula combining PoE with scope
allocation requires a new schema version rather than silently changing v1's
denominator. A scope's active total must be non-zero.

The initial implementation accepts exact domains only. Subtree scopes are a
separate, explicitly tested `subtree=true` extension; it cannot be inferred
from a prefix match because overlapping prefixes create hidden scope capture.

`ScopedMemoryV1` is not an alternate user API. It is the canonical deterministic
copy of the already accepted `MemorySubmit` data. Its content hash and identity
must agree with the normal Badger memory record before either state is written.
The consensus state contains no embedding; embeddings and search indexes are
locally rebuildable projections.

## 6. Transactions and authorization

v11.9 reuses the existing signed `GovPropose` transaction envelope for scope
formation. `OpScopeAction` is a new app-v20-gated governance operation whose
payload is the complete canonical next `ScopeRecordV1` template. This avoids a
second direct-mutation transaction family and keeps one immutable byte string
as both the voted decision and the applied authority.

The app-v20 HTTP path has two deliberately separate principals. The exact
operator-signed REST/dashboard request (method, path, body, block-time
freshness, and single-use 8-byte nonce) proves authorization. The signed body
also contains the validator ID and committed chain domain returned by
`GET /v1/governance/context`; consensus checks both before claiming the proof,
so neither a mempool observer nor another reachable validator can transplant or
burn it, and the same proof is invalid on a different SAGE chain. The outer
transaction key must be an active CometBFT validator and remains the proposal
owner, deterministic-ID input, automatic voter, later voter/canceller, and sole
holder of voting power. A reachable validator cannot be borrowed by an
arbitrary authenticated agent, and the built-in direct/no-proof upgrade voter
remains wire-compatible. Both `sage-gui` and `amid` must inject their concrete
private-validator key; governance fails closed when either that key or the
configured operator identity is missing.

Proposal authorization and vote ownership are intentionally different. Every
delegated post-app-v20 proposal operation requires the embedded operator to be
a registered global admin. Delegated vote/cancel requests require the exact
node-local operator proof plus the outer validator's normal voting/proposer
authority, but do not require all validators to share the global admin key.
Before activating app-v20, each validator that will expose a proposal gateway
must therefore register its proposal operator as a global admin. A legacy or
degenerate topology where only the validator key is an admin must either submit
the proposal directly with that validator key or materialize the operator's
admin registration before activation.

The minimal transaction family is:

| Operation | Effect | Authority |
| --- | --- | --- |
| `OpScopeAction` create | creates revision 1 with domains, members and weights | governance supermajority; proposer and members must be active validators |
| `OpScopeAction` update | adds/removes domains or members, changes weights, pauses/resumes | governance supermajority; revision advances by exactly one |
| `OpScopeAction` retire | permanently retires a scope and removes future mapping | governance supermajority over the exact retired record |
| `ScopeRecover` | installs a verified canonical-content chunk for an existing scoped memory when rebuilding a projection | only the deterministic block path; not a public remote-write escape hatch |

Group formation and every material roster change are governance-attested. A
single controller key is not sufficient. The proposal payload is the complete
next `ScopeRecordV1` template itself; its created/updated heights are zero in the
signed payload and are materialized from the deterministic quorum-execution
block. Execution re-verifies the exact accepted payload, current revision,
sorted membership, active validator eligibility, non-zero total weight,
registered exact domains, and no domain collision. There is no direct scope
mutation transaction or remote-write escape hatch.

The formation slice implements create, revision-checked update, pause/resume,
and permanent retire through `OpScopeAction`; it does not add a generic
free-form mutation endpoint. CEREBRUM and MCP may construct only canonical
payloads and must surface the scheduled governance decision rather than
claiming an immediate topology change.

## 7. Scoped memory lifecycle

At `postAppV20Fork(height)`:

1. `processMemorySubmit` performs the existing identity, RBAC, content, nonce,
   and domain checks first.
2. If the domain has no active scope mapping, the exact current lifecycle runs.
3. If it maps to an active scope, the app derives and stores a `ScopeBallotV1`
   snapshot and `ScopedMemoryV1` in the same deterministic block state change
   as the ordinary proposed memory. Any failure rejects the transaction; there
   is no partially mirrored scoped memory.
4. `processMemoryVote` accepts a scoped vote only from an active member pinned
   in that ballot. It uses the stored effective integer weights, not a live
   roster query. Nonmember votes are rejected rather than counted at zero.
5. A strict scoped supermajority commits. If every pinned member has cast a
   terminal vote without a supermajority, the memory becomes deprecated. A
   missing/offline member leaves the proposal pending; it is an availability
   event, not permission to lower the denominator.

Existing normal-memory vote records remain immutable. The scope ballot is a
supplementary record and no compatibility reader may mistake an unscoped memory
for a scoped one. The existing immediate-commit co-author envelope is rejected
for active or paused scoped domains in v1; supporting it requires a new signed
co-commit schema that explicitly binds the scope revision and ballot authority.

## 8. Membership changes, revocation, and recovery

- A member must be active in the on-chain validator set at scope-create/update
  execution. Removing that validator via the existing governance path requires
  a scope update or pauses every affected scope for future submissions.
- Removing a member is prospective: it creates a new revision for new ballots;
  older ballots retain their original proof denominator and may finish only
  under it. Before release, the operator UI must report this honestly as a
  drain requirement.
- A revoked member's future scoped vote and all new scoped submissions are
  rejected after the new revision. It cannot restore itself with a stale
  revision or by replaying an old `ScopeUpdate`.
- Ordered committed-block replay is the currently implemented catch-up path. An
  integration proof holds one replica offline, advances the other through a
  selected submit, an unselected submit, and scoped votes, then replays the
  exact blocks and compares every intermediate AppHash. Only the selected
  memory receives a canonical scoped envelope.
- Scoped votes persist their first FinalizeBlock height beside the immutable
  decision. If a projection flush fails after Badger was written but before
  committed height advances, the canonical scoped submit envelope or exact
  decision+height pair authorizes reconstruction of only that lost projection
  batch. A changed transaction or later height still fails the ordinary nonce
  guard. The integration proof exercises both submit and quorum-reaching vote
  crash windows across real store reopen cycles.
- On boot, a node validates every `ScopedMemoryV1` against its memory hash/status
  and rebuilds missing SQL rows plus clearance metadata transactionally.
  `/ready` fails closed while canonical scoped envelopes exist but that
  projection is missing or invalid; a locked SQLite vault retries the rebuild
  immediately after operator unlock, followed by normal serving-index repair.
- Legacy `agent_orgs` and `org_name` startup repair is bounded to 512 source
  rows and 1 MiB of derived writes per Badger transaction. Progress lives in
  checksummed, `0600`, atomically replaced sidecars inside the Badger directory,
  not in its logical keyspace: each batch commits and syncs derived rows before
  the sidecar cursor can advance. Startup scrubs the two exact marker keys used
  by short-lived v11.9 development builds before scanning, preserving the
  v11.8 AppHash. Logical Badger backup and Internet state sync omit sidecars;
  their absence safely replays idempotent index writes from the beginning.
  Automatic executable rollback durably invalidates both sidecars before the
  old binary restarts. An in-place manual downgrade to a binary predating these
  indexes is unsupported: restore that binary's matching rollback snapshot (or
  explicitly invalidate migration progress before returning to v11.9), rather
  than letting the old binary create unindexed rows behind a completion claim.
- The existing `internal/snapshot` bundle is **operator-local rollback only**.
  It includes SQLite, CometBFT databases, node and validator key files, optional
  vault material, config, and optionally the executable. It must never be
  advertised or transmitted as an ABCI/P2P snapshot. The local integration
  proof verifies the bundle, restores app-v20 and canonical scoped state,
  deletes SQLite completely, then rebuilds and re-verifies the projection.
- CometBFT ABCI state sync is fail-closed by default. An explicit mutually
  exclusive `quorum.state_sync.serving` or `receiving` boot role requires
  `quorum.enabled`, a strict locally installed JSON authorization, and the
  authorization's exact canonical Comet node-ID set. `provider_node_ids` must
  exactly equal `validator_node_ids`; the joining node is distinct, and
  `authorized_peer_ids` is exactly validators plus joiner. Serving/receiving
  endpoint controllers are armed before `StartChain`; missing/expired
  authorization or effective peer-profile drift fails startup.
- The in-process `sage-gui` path pins CometBFT v0.38.23 and locally hardens
  armed state-sync nodes: PEX, seeds and seed mode are disabled, normal inbound/
  outbound limits are zero, unconditional/private IDs exactly match the
  authorization for capacity/privacy, authenticated peer filtering is mandatory
  and admits only that exact ID set, and persistent peers are restricted to it;
  the receiver must persistently dial every existing validator/provider. The local dependency
  replacement pins exact upstream source commit
  `feb2aea4dc271d612129afc958cb844713ec792b` and overlays six files: the
  state-sync reactor's nil-syncer guard; a block-sync graceful-abort sentinel
  scoped only to SAGE seal shutdown; node startup that supplies and retains the
  effective-height bridge across consecutive empty-blockstore restarts;
  strict `seen commit -> positive state -> successful
  block-sync switch -> durable completion marker` ordering; atomic state plus
  effective-height-marker bootstrap; and synchronous/raw-database store
  handling. Exact lone seen-commit crash residue is removable only after the
  state DB is independently proven empty. Raw inspection occurs before a
  `BlockStore` is constructed, so additional or malformed metadata is preserved
  and rejected instead of triggering the upstream constructor panic. The
  standalone runtime reports the pinned source as
  `0.38.22+feb2aea4dc271d612129afc958cb844713ec792b` because the v0.38.23
  source tag still declares that core semver and SAGE stamps the exact commit.
  The receiver additionally requires two distinct HTTP(S) RPC
  origins plus trusted height/hash/period for Comet's light client.
  Because outbound peer addition and remote admission run on opposite
  processes, a rejected dialer can transiently list the locally approved
  provider before EOF. Ordinary provider inbound capacity is zero, so an
  unknown authenticated ID is rejected before `addPeer`; the wire gate also
  queries the live ID callback for exact rejected/approved codes `111/0`.
  The integrated gate requires the resulting authenticated provider-side
  rejection, repeatedly checks that the provider never adds that ID and the
  dialer obtains no snapshot, height, session, or REST service, drains both
  switches, and then requires the approved receiver to pass the callback.
- A provider maintains a dedicated snapshot root, exports its current committed
  app-v20 state as a deterministic latest-visible key/value stream, restores it
  into an isolated database to verify exact AppHash/scoped state, and atomically
  publishes only the consensus format before exposing P2P. The newest eight
  owned snapshots are retained; the latest two may await light blocks while six
  older fallback points remain. `ListSnapshots` exposes snapshot `H` only once
  live height is at least `H+2`; chunk loads use the same de-duplicated catalog
  and rehash on read. The format carries no SQL projection, Comet/node/validator
  key, vault material, local config, logs, binary, Badger physical versions, or
  tombstones. Provider startup also requires effective `retain_blocks=0`
  (`retain_blocks: 0` is the quorum default); a positive pruning window is
  rejected until rolling snapshots are block-base aware.
- A receiver is pristine only when application height/AppHash and raw Badger are
  empty, SQLite contains exactly the default domain seed rows and every other
  application table is empty, and persisted Comet state is empty. Startup may
  then remove only the exact lone valid seen-commit crash residue; afterward
  the `state`, `blockstore`, `evidence`, and `tx_index` databases must have no
  entries. The WAL is absent
  or a zero-length regular non-symlink file, FilePV signing state is zero, and
  genesis cannot make Comet skip state sync as a single-local-validator chain.
  Offer preflights `2 * snapshot + 256 MiB`; candidate preparation separately
  preflights `4 * snapshot + 256 MiB` on its target filesystem.
  Preflight capacity failures and native preparation exhaustion (`ENOSPC`,
  `EDQUOT`, and Windows disk-full/quota equivalents) are terminal local errors
  and are never charged to provider fallback. The receiver checks every chunk
  sender, assembles out of order with exact duplicate semantics, and
  pre-verifies a candidate in an isolated database
  while live state stays open. A bad candidate can fall back to another
  authorized snapshot, bounded to eight rejections.
- Receiver expiry is rechecked throughout offer, chunking, assembly,
  preverification, bundle activation, and each durable journal/rename boundary;
  provider expiry is rechecked on every list/load request. Both use a
  process-local monotonic deadline and permanently latch expiry or an observed
  backward clock step. Every new address/ID admission rechecks the same latch;
  expiry does not live-disconnect an already-established Comet connection, so
  operators must still replace the ceremony peer profile and restart.
  Successful seal and matching crash recovery automatically mutate
  only the raw `quorum.state_sync.receiving` YAML node to `false`, fsync that
  transition, and only then remove the recovery journal. Provider `serving`
  remains an explicit operator-controlled role.
- After successful preverification, activation synchronously installs a
  complete replacement application bundle before final `ACCEPT`. Ordinary ABCI
  consensus methods fail fast in discovery, assembly, and preparation; after
  Comet's block-sync switch, any early consensus block call waits without the
  bundle lease until seal. `Query` and `CheckTx` never wait: both fail fast in
  every unsealed phase, including PendingComet, so they cannot deadlock the
  syncer's final `Info` over Comet's shared in-process client mutex. Any local
  close, journal, rename, reopen, or bundle-activation
  failure from the `prepared` transition onward is terminal and never charged
  to provider semantic fallback. Startup then waits a bounded 30 minutes by
  default for exact Comet height/AppHash/app-version, a valid seen commit, and
  exact post-switch marker persistence, seals the journal/quarantine
  transaction, releases pending block execution, freezes the final app/store
  graph, and only then starts projection, REST/dashboard/MCP/federation, and
  background workers.
  Shutdown/failure releases a waiting block call with the SAGE seal-abort
  sentinel; only that sentinel exits block sync gracefully, while every
  unrelated application error retains upstream panic behavior.
- The detailed implementation and operator schema are in
  [`v11.9-state-sync-activation.md`](v11.9-state-sync-activation.md). Raw Comet
  P2P (normally TCP 26656) and at least two Comet RPC origins must be reachable;
  Internet validators still need routable addresses, port forwarding, or a VPN.
  The federation relay is not a consensus tunnel, and a future tunnel layer is
  outside v11.9. The integrated real `sage-gui serve` provider-to-receiver
  harness is present, but its final exact-source cold pass still blocks release;
  the split `amid`/Comet topology does not exercise whole-bundle activation or
  the seal boundary.
- Snapshot restore and a discarded local projection remain first-class tests.
  A node must never serve a claimed-committed scoped memory whose canonical
  mirror is absent or mismatched.

v11.9 does not promise cross-chain deletion, per-chain lifecycle convergence,
or a right to erase copies from pre-v11.9 federation peers.

## 9. Threat model and negative cases

| Threat | Required defence |
| --- | --- |
| Controller or dashboard compromise | cannot mutate a scope without a governance-approved exact record |
| Authenticated agent reaches another validator's HTTP port | gateway accepts only that node's configured operator; app-v20 binds the exact delegated action while governance power remains with the outer active validator |
| Mempool observer transplants or burns an operator proof | signed body names the exact outer validator and committed chain domain; consensus validates both before the single-use replay claim |
| Cloned request is replayed on another chain | app-v20 quorum commits a domain derived from the exact CometBFT chain ID; mismatch is rejected before proof claim |
| Stale roster/replay | revision predecessor, immutable revision hash, strict next-revision check, and AppHash replication |
| Validator removed mid-flight | ballot pins old membership; new scope writes pause until a prospective revision is committed |
| Weight manipulation | non-zero integer allocation, deterministic effective-weight conversion, stored ballot snapshot, exact integer threshold |
| Metadata leak from unselected domains | exact scope lookup, no global scope enumeration to federation peers, scope/domain collision rejection |
| Partial content loss after state sync | Badger canonical envelope + hash verification before projection/serving readiness |
| Off-consensus cache poisoning | Badger is authoritative; SQLite/PostgreSQL values are rebuilt or rejected on mismatch |
| Physical Badger/history leakage | wire format streams only latest-visible ordered keys; TTL/user metadata, physical versions, tombstones, and expired values are excluded or fail export |
| Local rollback bundle exfiltration | never expose it through ABCI/P2P; network state sync uses a separate key-free, config-free, consensus-only format |
| Expired join during activation | deadline-bound work, rechecks at durable boundaries, permanent receiver expiry latch, and rollback before seal |
| Disk exhaustion | size-bounded 2x/4x receiver preflights, native disk-full/quota classification as terminal local failure, owned-staging cleanup, and eight-snapshot provider retention |
| Stale pre-ceremony mempool entry | all-validator restart before bootstrap, advisory static/raw CheckTx ceilings, bounded Prepare/Finalize Code-111 draining, and an explicit 1-MiB external-Comet transaction cap |
| Network partition | CometBFT and the application denominator remain unchanged; no local override or quorum relaxation |
